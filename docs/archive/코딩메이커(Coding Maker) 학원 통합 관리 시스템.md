# [통합 개발 명세서] 코딩메이커(Coding Maker) 학원 통합 관리 시스템: 아키텍처 및 구현 전략 보고서

## 1. 서론: 프로젝트 정의 및 전략적 목표

### 1.1. 프로젝트 배경 및 정체성 확립

본 보고서는 전라남도 광양시 중동(무등길 47)에 위치한 '코딩메이커학원(Coding Maker Academy)'의 운영 효율화와 교육 서비스 고도화를 위한 통합 관리 시스템(ERP/LMS) 구축의 기술적 실행 계획을 상세히 기술한다. 본 프로젝트는 단순한 학원 홍보용 홈페이지 구축을 넘어, 임베디드/코딩, 웹툰/창작, 자격증 과정 등 다원화된 커리큘럼을 체계적으로 관리하고, 학생과 학부모, 강사 간의 유기적인 소통을 지원하는 올인원(All-in-One) 플랫폼 개발을 목표로 한다.1

특히 본 프로젝트는 일일 활성 사용자(DAU) 200명 이상의 트래픽을 안정적으로 처리해야 하며, 개인정보 보호를 위한 강력한 보안 아키텍처와 소규모 클래스 집중 관리에 최적화된 사용자 경험(UX)을 제공해야 한다. 운영 주체인 코딩메이커학원의 실질적인 교육 철학인 '메이커 교육'과 '실무 중심 코딩'을 디지털 환경으로 확장하는 것이 핵심이며, 이는 곧 광양 지역 내 IT 교육 거점으로서의 입지를 공고히 하는 디지털 전환(Digital Transformation) 전략의 일환이다.

### 1.2. 핵심 운영 목표 및 요구사항 분석

시스템 구축의 핵심 목표는 파편화된 학원 행정 업무의 통합과 자동화에 있다. 기존의 오프라인 중심 출석 관리, 수동적인 상담 기록, 그리고 지류 기반의 안내문 발송 방식은 데이터의 분절화를 초래하고 운영 리소스를 과다하게 소모시키는 원인이 되어왔다. 따라서 본 시스템은 다음과 같은 구체적인 운영 목표를 달성해야 한다.

첫째, **데이터 사일로(Data Silo) 제거 및 통합 관리**이다. 학생의 등하원 기록, 수강 중인 강좌의 진도율, 과제 제출 현황, 그리고 학부모 상담 이력이 하나의 데이터베이스 내에서 유기적으로 연결되어야 한다. 이를 통해 강사는 학생의 학습 패턴을 종합적으로 분석하여 개인 맞춤형 지도를 수행할 수 있다.4

둘째, **군사 등급 수준의 개인정보 보안성 확보**이다. 학원 시스템 특성상 미성년 학생들의 이름, 연락처, 거주지 주소 등 민감한 개인정보(PII)를 다수 취급하게 된다. 따라서 데이터베이스가 물리적으로 탈취되거나 유출되는 최악의 보안 사고 상황에서도, 암호화 키 없이는 데이터의 내용을 절대 식별할 수 없도록 애플리케이션 레벨에서의 강제적인 암호화(AES-256)가 구현되어야 한다.6

셋째, **유연하고 확장 가능한 커리큘럼 관리(CMS)**이다. 코딩 교육 시장은 트렌드 변화가 매우 빠르다. 따라서 개발자의 개입 없이도 관리자가 관리자 페이지(Admin Dashboard)를 통해 'AI 활용반', '유니티 게임 제작반' 등의 신규 과정을 즉시 개설하고, 커리큘럼 내용을 수정할 수 있는 동적 컨텐츠 관리 시스템이 필수적이다.8

## 2. 시스템 아키텍처 및 기술 스택 설계

### 2.1. 핵심 기술 스택 선정 근거

본 프로젝트는 최신 웹 기술 트렌드를 반영하면서도 장기적인 유지보수성과 안정성을 보장하기 위해 다음과 같은 기술 스택을 선정하였다.

|**구분**|**기술명**|**선정 근거 및 특징**|
|---|---|---|
|**프레임워크**|Next.js 14+ (App Router)|React 서버 컴포넌트(RSC)를 통한 초기 로딩 속도 최적화 및 SEO 강화. 서버 사이드 렌더링(SSR)과 정적 생성(SSG)의 하이브리드 지원.|
|**런타임**|Node.js (LTS)|방대한 생태계와 비동기 I/O 처리에 최적화된 성능. Socket.io와의 높은 호환성.|
|**언어**|TypeScript|정적 타입 검사를 통한 런타임 에러 사전 방지 및 코드 안정성 확보. 대규모 프로젝트 협업 용이성.|
|**스타일링**|Tailwind CSS|유틸리티 퍼스트 접근 방식을 통한 빠른 UI 개발 및 유지보수 용이성. 디자인 시스템(Design Token) 적용 수월.|
|**ORM**|Prisma|직관적인 스키마 정의 및 타입 안전성이 보장된 쿼리 빌더 제공. 마이그레이션 자동화 도구 지원.|
|**상태 관리**|Zustand / TanStack Query|클라이언트 전역 상태(세션, 소켓) 관리를 위한 경량 라이브러리(Zustand)와 서버 데이터 캐싱(TanStack Query)의 분리 적용.|

### 2.2. 커스텀 서버(Custom Server) 전략과 Socket.io 통합

본 프로젝트의 가장 큰 기술적 도전 과제 중 하나는 Next.js의 App Router 구조 내에서 실시간 양방향 통신을 위한 Socket.io를 안정적으로 통합하는 것이다. Next.js는 기본적으로 서버리스(Serverless) 환경을 지향하므로, 상태를 유지해야 하는 웹소켓 연결과는 구조적인 충돌이 발생할 수 있다.10

#### 2.2.1. 아키텍처적 딜레마와 해결 방안

일반적인 Vercel 배포 환경에서는 서버리스 함수가 요청 처리 후 즉시 종료되므로 지속적인 소켓 연결을 유지할 수 없다. 이를 해결하기 위해 외부 서비스(Pusher 등)를 사용할 수도 있으나, 비용 효율성과 데이터 보안(자체 서버 내 데이터 처리)을 위해 **커스텀 서버(Custom Server) 아키텍처**를 채택한다.

이 구조에서는 `server.ts`라는 별도의 진입점 파일을 생성하여 Node.js의 기본 `http` 서버를 초기화하고, 이 서버 인스턴스를 Next.js의 요청 핸들러와 Socket.io 서버가 공유하도록 구성한다. 이는 단일 포트에서 HTTP 요청과 WebSocket 연결을 동시에 처리할 수 있게 하여 CORS(Cross-Origin Resource Sharing) 문제를 최소화하고 배포 복잡도를 낮춘다.10

#### 2.2.2. 배포 환경의 제약 사항

커스텀 서버를 사용함에 따라 Vercel과 같은 서버리스 전용 플랫폼으로의 배포는 불가능해진다. 대신, Docker 컨테이너 기반의 배포나 VPS(Virtual Private Server) 환경(예: AWS EC2, Cafe24 가상서버)을 사용해야 한다. 이는 'Automatic Static Optimization'과 같은 Next.js의 일부 최적화 기능을 비활성화시키지만, 실시간 출석 알림과 채팅 기능의 안정성을 위해 감수해야 할 기회비용으로 판단된다.10

#### 2.2.3. 소켓 인스턴스 관리 (Monkey Patching)

Next.js의 개발 환경(Hot Reloading)에서는 코드 수정 시마다 서버가 재시작되면서 소켓 인스턴스가 중복 생성되는 문제가 발생할 수 있다. 이를 방지하기 위해 전역 객체(`global` 또는 `res.socket.server.io`)에 소켓 인스턴스를 주입하여 싱글톤(Singleton) 패턴을 모방하는 'Monkey Patching' 기법을 적용한다. 이를 통해 API 라우트 내에서도 이미 생성된 소켓 인스턴스에 접근하여 이벤트를 발송(Emit)할 수 있는 구조를 완성한다.13

### 2.3. 이중 데이터베이스(Dual DB) 운용 전략

#### 2.3.1. 환경별 데이터베이스 분리

개발 속도와 운영 안정성이라는 두 마리 토끼를 잡기 위해 이중 데이터베이스 전략을 수립한다. 로컬 개발 환경에서는 설정이 간편하고 파일 기반으로 작동하는 **SQLite**를 사용하며, 실제 운영(Production) 환경에서는 동시 접속 처리 능력과 데이터 무결성이 뛰어난 **MySQL(또는 MariaDB)**를 사용한다.15

|**환경**|**데이터베이스**|**선정 이유**|**연결 방식**|
|---|---|---|---|
|**Development**|SQLite (`file:./dev.db`)|별도의 서버 설치 불필요, 빠른 프로토타이핑, 파일 공유를 통한 협업 용이.|로컬 파일 직접 액세스|
|**Production**|MySQL (v8.0+)|트랜잭션 처리 성능 우수, 동시성 제어(Locking), 데이터 복구 및 백업 용이성.|TCP/IP 소켓 연결|

#### 2.3.2. Prisma 스키마의 동적 스위칭 구현

Prisma ORM은 `schema.prisma` 파일 내의 `provider` 필드를 환경 변수로 동적으로 변경하는 기능을 기본적으로 지원하지 않는다.15 따라서 이를 해결하기 위해 빌드 시점에 환경 변수(`DATABASE_PROVIDER`)를 감지하여 스키마 파일을 재작성하는 전처리 스크립트(`scripts/switch-db.js`)를 도입한다.

이 스크립트는 정규 표현식을 사용하여 `provider = "sqlite"` 부분을 운영 환경에 맞게 `provider = "mysql"`로 치환한 후, `prisma generate` 명령어를 실행하여 클라이언트를 재생성한다. 이는 CI/CD 파이프라인 내에서 자동으로 수행되어 개발자의 실수로 인한 오류를 방지한다.15

## 3. 보안 아키텍처 및 암호화 전략

### 3.1. 데이터 프라이버시 원칙: Zero-Trust

본 시스템의 보안 설계는 "데이터베이스 관리자조차도 사용자의 민감 정보를 열람할 수 없어야 한다"는 Zero-Trust 원칙에 기반한다. 이를 위해 데이터베이스에 저장되기 직전에 애플리케이션 레벨에서 암호화를 수행하며, 복호화 역시 데이터를 조회하여 메모리에 로드하는 순간에만 수행된다.6

### 3.2. AES-256 양방향 암호화 구현 상세

민감 정보(이름, 전화번호, 주소, 상담 내용)의 보호를 위해 미 국립표준기술연구소(NIST)가 승인한 AES-256-CBC 알고리즘을 사용한다.

#### 3.2.1. 키(Key) 및 초기화 벡터(IV) 관리

암호화의 강도는 키 관리와 초기화 벡터(IV)의 무작위성에 달려 있다.

- **암호화 키(Cipher Key):** 256비트(32바이트) 길이의 고정된 키를 `.env` 파일의 `CIPHER_KEY` 환경 변수로 관리한다. 이 키는 소스 코드에 절대 하드코딩되어서는 안 되며, 운영 서버의 환경 변수 설정 기능을 통해서만 주입된다.
    
- **초기화 벡터(IV):** 동일한 평문(예: "홍길동")이라도 암호화할 때마다 매번 다른 암호문을 생성하기 위해, 매 암호화 시도 시 `crypto.randomBytes(16)`을 통해 16바이트의 고유한 IV를 생성한다.7 고정된 IV를 사용할 경우 패턴 분석을 통한 해독이 가능해지므로 이는 보안상 치명적이다.
    

#### 3.2.2. 데이터 저장 포맷 및 처리 로직

데이터베이스의 필드(문자열 타입)에는 IV와 암호문을 결합한 형태로 저장한다.

- **저장 형식:** `iv(hex):ciphertext(hex)` (예: `a1b2...:c3d4...`)
    
- **Prisma 확장(Extension) 적용:** 개발자가 매번 수동으로 암호화/복호화 함수를 호출하는 번거로움을 없애고 휴먼 에러를 방지하기 위해, Prisma Client Extension 기능을 활용한다. 이 확장은 `User` 모델의 데이터를 읽거나(`findUnique`, `findMany`) 쓸 때(`create`, `update`), 지정된 필드(`name`, `phone` 등)를 자동으로 감지하여 투명하게 암호화 및 복호화 로직을 수행한다.6
    

### 3.3. 접근 제어 및 권한 관리 (RBAC)

사용자 권한은 계층형(Tier System) 구조로 설계하여 관리의 효율성을 높인다.

- **Tier 1 (최고 관리자):** 시스템 전체 설정, 강사 계정 생성, 급여 및 매출 관리 접근 권한.
    
- **Tier 2 (강사):** 배정된 클래스의 출석 관리, 커리큘럼 해금(Unlock), 학생 상담 일지 작성 권한.
    
- **Tier 3 (학생/학부모):** 본인(또는 자녀)의 출석 현황 조회, 과제 제출, 공지사항 열람 권한. 학부모 계정(Tier 3-B)은 자녀 계정과 외래키(`parentId`)로 연결되어 읽기 전용 권한을 갖는다.
    
- **Middleware:** Next.js의 `middleware.ts`에서 JWT 토큰 내의 `role` 및 `tier` 정보를 검사하여, 권한이 없는 페이지 접근 시 즉시 리디렉션 처리한다.
    

## 4. 데이터베이스 스키마 설계 및 데이터 모델링

### 4.1. Prisma Schema 상세 정의

본 시스템의 데이터 모델은 사용자의 다층적 관계와 교육 과정의 유연성을 고려하여 설계되었다.

#### 4.1.1. 사용자(User) 모델

사용자 모델은 학생, 학부모, 강사, 관리자를 모두 포괄하며, 자기 참조 관계(Self-relation)를 통해 학부모와 자녀를 연결한다.

코드 스니펫

```
model User {
  id            String    @id @default(uuid())
  username      String    @unique // 로그인 ID
  password      String    // bcrypt 해시값 (Salt 포함)
  
  // 암호화 필수 필드 (저장 포맷: iv:ciphertext)
  name          String    // AES-256 암호화
  phone         String?   // AES-256 암호화 (연락처)
  address       String?   // AES-256 암호화 (거주지)
  
  tag           String?   @unique // 출석용 태그 ID (RFID 또는 키패드 번호, 예: #1234)
  tier          Int       @default(4) // 1:원장, 2:강사, 3:학생/학부모, 4:대기, 5:게스트
  role          String    @default("STUDENT") // ENUM 대체 (SQLite 호환성 고려)
  
  // 학부모-자녀 관계 (Self-Relation)
  parentId      String?
  parent        User?     @relation("ParentChild", fields: [parentId], references: [id])
  children      User    @relation("ParentChild")
  
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  
  // 관계 정의
  attendance    Attendance
  posts         Post
  sentMessages  Message @relation("SentMessages")
  chatRooms     ChatRoomMember
  progress      CourseProgress
}
```

#### 4.1.2. 커리큘럼(Course) 및 학습 관리(LMS) 모델

동적 커리큘럼과 드립 콘텐츠(Drip Content) 기능을 지원하기 위해 계층적 구조(Course -> Section -> Lesson)를 채택한다.4

코드 스니펫

```
model Course {
  id          String   @id @default(uuid())
  title       String   // 예: "임베디드 전문가 과정"
  category    String   // CODING, MAKER, CERTIFICATION
  description String?  @db.Text // HTML/Markdown 설명
  instructor  String?  // 담당 강사명 (비정규화 또는 User ID 참조)
  price       Int?     // 수강료 (예: 150,000원) 
  durationMin Int?     // 총 수업 시간 (예: 1000분) 
  isActive    Boolean  @default(true)
  
  sections    Section
  createdAt   DateTime @default(now())
}

model Section {
  id          String   @id @default(uuid())
  courseId    String
  course      Course   @relation(fields: [courseId], references: [id], onDelete: Cascade)
  title       String   // 예: "1주차: C언어 기초"
  order       Int      // 정렬 순서
  lessons     Lesson
}

model Lesson {
  id          String   @id @default(uuid())
  sectionId   String
  section     Section  @relation(fields: [sectionId], references: [id], onDelete: Cascade)
  title       String   // 예: "변수와 자료형"
  content     String   @db.Text // 강의 내용, 비디오 임베드
  
  // Drip Content 설정
  unlockDay   Int      @default(0) // 등록일 기준 X일 후 공개
  
  progress    CourseProgress
}
```

## 5. 핵심 기능 구현 상세 및 비즈니스 로직

### 5.1. 키오스크 모드 및 스마트 출석 시스템

출석 관리는 `/kiosk` 경로에서 제공되는 전체 화면 인터페이스를 통해 이루어진다. 이 페이지는 별도의 인증 없이(또는 IP 화이트리스트 제한 하에) 접근 가능하며, 터치스크린 환경에 최적화된 대형 숫자 키패드를 UI로 제공한다.

#### 5.1.1. 등하원 처리 로직 (Debounce & Toggle)

사용자가 자신의 고유 태그 번호(예: 1234)를 입력하면 다음과 같은 로직이 수행된다.

1. **사용자 식별:** 입력된 태그 번호와 매칭되는 활성 사용자(`User`)를 검색한다.
    
2. **중복 입력 방지(Debounce):** 해당 사용자의 마지막 출석 기록이 5분 이내에 존재한다면, 실수에 의한 중복 입력으로 간주하여 무시하거나 경고 메시지를 띄운다.
    
3. **상태 토글(Toggle):**
    
    - 오늘 날짜의 출석 기록이 없으면 → **등원(ARRIVAL)** 처리.
        
    - 이미 등원 기록이 있고 하원 기록이 없으면 → **하원(DEPARTURE)** 처리.
        
4. **피드백:** 화면에 "홍길동님, 안녕하세요!" 또는 "안녕히 가세요!"라는 메시지가 표시된다.
    

#### 5.1.2. 실시간 학부모 알림 발송

출석 데이터가 DB에 커밋(Commit)되는 즉시, Socket.io 서버는 해당 학생과 연결된 학부모 계정(`parentId`)을 조회한다. 학부모가 앱(또는 웹)에 접속해 있다면 실시간 푸시 알림을 전송하며, 접속하지 않은 상태라면 카카오톡 알림톡 API(추후 연동) 등을 통해 메시지를 발송할 수 있는 트리거를 남긴다.

### 5.2. 동적 커리큘럼 및 드립 콘텐츠(Drip Content)

광양 코딩메이커학원의 특성상 '컴퓨터그래픽스', '하드코딩', '자격증' 등 다양한 과정이 운영되며, 수강 기간이 24개월을 초과하는 장기 과정도 존재한다.1 이를 효과적으로 관리하기 위해 드립 콘텐츠 시스템을 도입한다.

#### 5.2.1. 시간차 공개(Time-Lock) 메커니즘

관리자는 각 레슨별로 `unlockDay`를 설정할 수 있다. 예를 들어, 'C언어 포인터' 수업의 `unlockDay`를 14로 설정하면, 학생이 해당 강좌에 등록(`Enrollment`)한 날로부터 14일이 지나야만 해당 콘텐츠에 접근할 수 있다.

- **로직:** `if (CurrentDate >= EnrollmentDate + Lesson.unlockDay) { Access = true } else { Access = false, Show "D-day" }`
    
- **효과:** 학생이 기초 과정을 건너뛰고 어려운 내용으로 바로 넘어가는 것을 방지하여 체계적인 학습을 유도하고, 중도 포기를 줄이는 효과가 있다.5
    

### 5.3. 실시간 상담 및 채팅 시스템

Socket.io 기반의 실시간 채팅은 학생의 질의응답과 학부모 상담 창구로 활용된다.

#### 5.3.1. 보안 채팅방 아키텍처

모든 채팅 메시지는 전송 단계부터 암호화되어야 한다.

1. **클라이언트:** 메시지 전송 시 Socket 이벤트를 발생시킨다 (`socket.emit('send_msg', { content, roomId })`).
    
2. **서버:** 수신된 메시지(`content`)를 `crypto.ts` 유틸리티를 사용해 즉시 암호화(`iv:ciphertext`)한 후 `Message` 테이블에 저장한다.
    
3. **브로드캐스트:** 같은 방(`roomId`)에 접속한 다른 클라이언트들에게 메시지를 전파할 때도 암호화된 상태로 전송하거나, 서버 메모리 상에서 복호화하여 전송할 수 있다. 본 시스템에서는 **종단간 암호화(E2EE)**에 준하는 보안을 위해, 클라이언트에게 암호화된 데이터를 보내고 클라이언트 측에서(세션 키 등을 활용하여) 복호화하는 방식을 권장하나, 구현 복잡도를 고려하여 서버에서 복호화 후 전송하는 방식을 1차적으로 채택한다. 단, DB에는 반드시 암호화되어 저장된다.
    

### 5.4. 관리자용 CMS 및 데이터 시각화

최고 관리자(원장)를 위한 대시보드는 학원 운영의 관제탑 역할을 한다.

#### 5.4.1. 학원 정보 동적 관리

`AcademyInfo` 테이블을 수정할 수 있는 폼(Form) 인터페이스를 제공한다. 원장이 전화번호를 변경하거나, 공지사항 배너 문구를 수정하면, 별도의 코드 배포 없이 웹사이트의 Footer나 메인 화면에 즉시 반영된다. 초기 데이터(Seed)로는 실제 운영 정보인 '061-745-3355', '전남 광양시 무등길 47' 등이 설정된다.1

#### 5.4.2. 데이터 시각화 (Recharts)

축적된 출석 및 성적 데이터를 시각화하여 의사결정을 지원한다.

- **출석률 그래프:** 일별/월별 등원 인원 추이를 꺾은선 그래프로 표시.
    
- **수강 과목 분포:** 코딩/메이커/자격증 수강생 비율을 파이 차트로 표시하여 마케팅 전략 수립에 활용.


### 6.2. 대량 등록 기능 (Excel Upload)

신규 학기 시작 시 수십 명의 학생 정보를 일일이 입력하는 것은 비효율적이다. 이를 해결하기 위해 엑셀 업로드 기능을 구현한다.

- **라이브러리:** `exceljs` 또는 `xlsx` (SheetJS)를 사용한다. `exceljs`는 스타일링이 포함된 템플릿 생성에 유리하며, 단순 데이터 파싱에는 `xlsx`가 성능상 유리하다.21 본 프로젝트는 관리자가 다운로드할 '입력 양식(Template)'의 서식(셀 색상, 유효성 검사 등)이 중요하므로 **`exceljs`**를 채택한다.
    
- **프로세스:** 업로드된 엑셀 파일의 행(Row)을 순회하며, 데이터 유효성을 검증(전화번호 형식 확인 등)하고, 비밀번호 해싱 및 개인정보 암호화를 수행한 뒤 `prisma.$transaction`을 통해 일괄 등록한다.
    

### 6.3. 배포 및 인프라 구성 (DevOps)

커스텀 서버와 Socket.io 사용으로 인해 Vercel 배포가 불가능하므로, 다음과 같은 인프라 구성을 권장한다.

- **서버:** AWS Lightsail 또는 Cafe24 가상서버 (Ubuntu 22.04 LTS).
    
- **프로세스 관리:** PM2 (Process Manager 2)를 사용하여 Node.js 프로세스를 데몬화하고, 서버 재부팅 시 자동 실행되도록 설정한다.
    
- **리버스 프록시:** Nginx를 앞단에 배치하여 SSL(HTTPS) 인증서를 적용하고, `/socket.io` 경로로 들어오는 요청을 Node.js 서버 포트(예: 3000)로 포워딩한다. WebSocket 업그레이드 헤더 처리가 필수적이다.
    

## 7. 결론 및 제언

본 통합 개발 명세서는 광양 코딩메이커학원의 운영 효율성을 극대화하고, 학생 및 학부모에게 수준 높은 교육 서비스를 제공하기 위한 기술적 청사진을 제시하였다.

**커스텀 서버 아키텍처**를 통해 Next.js의 강력한 렌더링 성능과 Socket.io의 실시간성을 결합함으로써, 단순한 정보 제공 사이트를 넘어선 인터랙티브 플랫폼을 구축할 수 있다. 또한, **이중 데이터베이스 전략**과 **자동화된 스키마 스위칭**은 개발 생산성과 운영 안정성이라는 상충되는 가치를 조화롭게 해결하는 방안이다.

무엇보다 **AES-256 기반의 철저한 암호화 전략**은 교육 기관으로서 지켜야 할 윤리적, 법적 책임을 기술적으로 보증하며, 학부모들에게 깊은 신뢰를 줄 수 있는 핵심 요소이다. 마지막으로, **동적 커리큘럼 관리(CMS)** 기능은 급변하는 IT 교육 환경에서 학원이 유연하게 대처하고 지속적으로 성장할 수 있는 기반이 될 것이다.

본 시스템이 성공적으로 구축된다면, 코딩메이커학원은 지역 내 독보적인 '스마트 에듀테크(Smart Edutech)' 교육 기관으로 자리매김할 것이며, 축적된 교육 데이터는 향후 AI 기반의 맞춤형 학습 추천 시스템으로 발전할 수 있는 귀중한 자산이 될 것이다.
